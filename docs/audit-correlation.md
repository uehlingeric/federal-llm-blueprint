# Audit Correlation: Linking LLM Requests Across Logging Planes

When a client calls the LLM gateway to process a document and invoke Bedrock, the request flows through three independent logging planes. This document shows how to correlate one end-to-end request and what the correlation limits are.

---

## The Three Logging Planes

### Plane (a): ALB Access Logs (S3)

The application load balancer logs every HTTPS request in the standard ALB access-log format. Location: the `alb-logs` S3 bucket under the `alb/` prefix (created by modules/document-store).

**Key fields (ALB access-log format):**
- `type` – listener type (`https` or `h2` on this internal TLS listener)
- `time` – time the load balancer generated the response, ISO 8601 (example: `2026-07-04T14:35:24.573456Z`)
- `elb` – load balancer name (e.g., `fedllm-dev-gateway`)
- `client_ip` – source IP (internal VPC only)
- `client_port` – source port
- `target_ip` – ECS task IP
- `target_port` – container port (e.g., 4000)
- `request_processing_time` – seconds before the ALB sent the request to the target
- `target_processing_time` – seconds while the target processed (for an LLM completion this is the dominant term)
- `response_processing_time` – seconds after the target replied
- `elb_status_code` – HTTP status from ALB to client
- `target_status_code` – HTTP status from container
- `received_bytes`, `sent_bytes` – payload sizes
- `request` – HTTP verb, path, HTTP/1.1 or HTTP/2 (e.g., `POST /api/chat HTTP/2.0`)
- `user_agent` – client user agent
- `ssl_cipher` – TLS cipher suite
- `ssl_protocol` – TLS version (e.g., TLSv1.3)
- `target_group_arn` – target group of the request
- `trace_id` – **Trace ID**: the `X-Amzn-Trace-Id` header forwarded from the client, in format `Root=1-<epoch-hex>-<random>` (e.g., `Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f`)
- `domain_name` – SNI hostname
- `chosen_cert_arn` – certificate used
- `matched_rule_priority` – ALB listener rule matched

**Format example:**

```
https 2026-07-04T14:35:24.573456Z fedllm-dev-gateway 10.0.1.42 50123 10.0.2.15 4000 0.001 2.450 0.000 200 200 512 1024 POST https://fedllm-dev-gateway.internal:443/api/chat HTTP/2.0 curl/7.85.0 ECDHE-RSA-AES256-GCM-SHA384 TLSv1.3 arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/gw*/18c4a8b5c7d2e9f1 "Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f" "-" "-" "-" "-" "-" "-"
```

### Plane (b): Gateway Container Logs (CloudWatch Logs)

The ECS task runs LiteLLM, which logs each request as it arrives and processes. Location: CloudWatch Logs group `/ecs/fedllm-dev-gateway` (or `/ecs/{project}-{environment}-gateway` in the general case), created by modules/ecs-llm-gateway.

**How the trace ID gets here:** The ALB passes the `X-Amzn-Trace-Id` header to the target (the ECS task). LiteLLM can be configured to emit request logs that include this header (and other context). The exact fields LiteLLM logs depend on its logging configuration. **This example is ILLUSTRATIVE:**

```
[2026-07-04 14:35:22] LiteLLM Request Start - request_id=1a2b3c4d5e6f7a8b9c0d1e2f, model=anthropic.claude-sonnet-4-5-20250929-v1:0, trace_id=Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f, user_role=arn:aws:iam::123456789012:role/fedllm-dev-app, tokens_in=250
[2026-07-04 14:35:22.045] LiteLLM Request End - request_id=1a2b3c4d5e6f7a8b9c0d1e2f, status=200, tokens_out=180, latency_ms=45
```

In reality, LiteLLM logs depend on the logging level and format specified in the gateway's config.yaml. Check the LiteLLM documentation for the exact fields. The critical point: if `X-Amzn-Trace-Id` is included in the log line, it can link to plane (a).

### Plane (c): Bedrock Model-Invocation Logs (CloudWatch Logs)

Bedrock logs each model invocation in CloudWatch Logs. Location: Log group `/aws/fedllm-dev/bedrock-invocations` (or `/aws/{project}-{environment}/bedrock-invocations`), stream `aws/bedrock/modelinvocations`, created by modules/audit.

**Fields (from the Bedrock invocation-logging schema):**
- `schemaType` – "ModelInvocationLog"
- `schemaVersion` – version string (e.g., "1.0")
- `timestamp` – UTC time in ISO 8601 format (e.g., `2026-07-04T14:35:24Z`)
- `accountId` – AWS account ID (e.g., `123456789012`)
- `region` – AWS region (e.g., `us-east-1`)
- `requestId` – **Bedrock Request ID**: UUID generated by Bedrock for this call (e.g., `3f8c1a9b-2d4e-4f61-8a9b-0c1d2e3f4a5b`)
- `operation` – e.g., "InvokeModel", "InvokeModelWithResponseStream", "Converse", "ConverseStream"
- `modelId` – e.g., `anthropic.claude-sonnet-4-5-20250929-v1:0`
- `identity.arn` – IAM principal ARN of the caller (e.g., `arn:aws:iam::123456789012:role/fedllm-dev-app`)
- `input.inputTokenCount` – tokens sent to the model
- `output.outputTokenCount` – tokens returned by the model

**Metadata-only example (enable_full_content_logging = false):**

```json
{
  "schemaType": "ModelInvocationLog",
  "schemaVersion": "1.0",
  "timestamp": "2026-07-04T14:35:24Z",
  "accountId": "123456789012",
  "region": "us-east-1",
  "requestId": "3f8c1a9b-2d4e-4f61-8a9b-0c1d2e3f4a5b",
  "operation": "InvokeModel",
  "modelId": "anthropic.claude-sonnet-4-5-20250929-v1:0",
  "identity": {
    "arn": "arn:aws:iam::123456789012:role/fedllm-dev-app"
  },
  "input": {
    "inputTokenCount": 250
  },
  "output": {
    "outputTokenCount": 180
  }
}
```

## Correlation Keys and Their Limits

### Plane (a) → Plane (b): ALB Trace ID

**Correlation key:** `X-Amzn-Trace-Id` header value (e.g., `Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f`)

**Where to find it:**
- Plane (a): ALB access logs, `trace_id` field
- Plane (b): Container logs, if LiteLLM is configured to emit it (not guaranteed; check your config.yaml)

**Limitations:**
- Requires LiteLLM to forward and log the header.
- One ALB request may trigger multiple model invocations (e.g., a single user prompt → multiple Bedrock calls for parallelized summarization). The ALB log shows one client request; plane (b) shows N container-internal calls.

### Plane (b) → Plane (c): Timestamp + Identity + Model

**Correlation keys:**
- `timestamp` (within a 1–2 second window to account for clock skew)
- `identity.arn` (the IAM principal)
- `modelId` (which model was invoked)

**Where to find them:**
- Plane (b): Container log timestamps bound the Bedrock call window; the invoking principal is known from the deployment (all gateway traffic invokes Bedrock as the app task role)
- Plane (c): Bedrock logs emit timestamp, identity.arn, and modelId

**Important limitation:** There is **NO shared request ID** between planes (b) and (c). Bedrock generates its own `requestId` (unique per Bedrock call), which does NOT appear in plane (b). Correlation across this seam is **probabilistic under concurrency**: if the task role invokes Bedrock once per client request, timestamp + identity + model is sufficient; if the task invokes Bedrock twice in rapid succession (< 2 seconds), you cannot definitively separate which invocation produced which log in plane (c) without additional context (e.g., model input size from plane (b) to match inputTokenCount in plane (c)).

For **deterministic** correlation under concurrency, the application code must:
1. Capture Bedrock's `requestId` from the API response in plane (b), or
2. Include a correlation UUID in the prompt context (as a comment) so that you can search plane (c) by the same UUID in the full-content logs.

This is a known gap when full-content logging is not enabled. Document this in your runbook.

## Worked Trace: One Request Through Three Planes

**Scenario:** At 2026-07-04 14:35:22 UTC, a RAG application in the VPC (which has already retrieved its document context from S3 and the vector store) calls the gateway's OpenAI-compatible completions endpoint. The gateway (LiteLLM) routes the completion to Bedrock. The gateway is a model proxy only — document retrieval happens in the application layer and appears in the S3 data-event trail, not in these planes.

### Plane (a): ALB Log

```
https 2026-07-04T14:35:24.573456Z fedllm-dev-gateway 10.0.1.42 50123 10.0.2.15 4000 0.001 2.450 0.000 200 200 512 1024 POST https://fedllm-dev-gateway.internal:443/api/chat HTTP/2.0 curl/7.85.0 ECDHE-RSA-AES256-GCM-SHA384 TLSv1.3 arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/gw*/18c4a8b5c7d2e9f1 "Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f" "-" "-" "-" "-" "-" "-"
```

**Trace ID:** `Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f`  
**Timestamp:** 2026-07-04T14:35:24.573456Z (response generated)  
**Status:** 200 (success)  
**Latency:** 2.450 s (target processing — the Bedrock completion dominates)

### Plane (b): Container Log (ILLUSTRATIVE — TRANSCRIPT PENDING)

```
[2026-07-04 14:35:22.120] LiteLLM Request Start - request_id=req-xyz-123, trace_id=Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f, endpoint=/v1/chat/completions
[2026-07-04 14:35:22.135] LiteLLM Bedrock Call Start - model=anthropic.claude-sonnet-4-5-20250929-v1:0, prompt_tokens=250
[2026-07-04 14:35:24.540] LiteLLM Bedrock Call End - bedrock_request_id=3f8c1a9b-2d4e-4f61-8a9b-0c1d2e3f4a5b, response_tokens=180
[2026-07-04 14:35:24.560] LiteLLM Request End - request_id=req-xyz-123, status=200, total_s=2.44
```

**Trace ID:** `Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f` (matches plane a)  
**Timestamp:** 2026-07-04 14:35:22.135 (Bedrock call start)  
**IAM Role:** `arn:aws:iam::123456789012:role/fedllm-dev-app`  
**Model:** `anthropic.claude-sonnet-4-5-20250929-v1:0`  
**Bedrock Request ID (from response):** `3f8c1a9b-2d4e-4f61-8a9b-0c1d2e3f4a5b`

### Plane (c): Bedrock Invocation Log

```json
{
  "schemaType": "ModelInvocationLog",
  "schemaVersion": "1.0",
  "timestamp": "2026-07-04T14:35:24Z",
  "accountId": "123456789012",
  "region": "us-east-1",
  "requestId": "3f8c1a9b-2d4e-4f61-8a9b-0c1d2e3f4a5b",
  "operation": "InvokeModel",
  "modelId": "anthropic.claude-sonnet-4-5-20250929-v1:0",
  "identity": {
    "arn": "arn:aws:iam::123456789012:role/fedllm-dev-app"
  },
  "input": {
    "inputTokenCount": 250
  },
  "output": {
    "outputTokenCount": 180
  }
}
```

**Timestamp:** 2026-07-04T14:35:24Z (matches the plane-b Bedrock call window)  
**Request ID:** `3f8c1a9b-2d4e-4f61-8a9b-0c1d2e3f4a5b` (captured from Bedrock response in plane b)  
**Identity:** `arn:aws:iam::123456789012:role/fedllm-dev-app` (matches the task role in plane b)

### Correlation Summary

| Plane | Time (UTC) | Trace ID / Request ID | Key Details |
|-------|---|---|---|
| (a) ALB | 14:35:24.573 (response) | `Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f` | HTTP 200, 2.450 s target processing |
| (b) Gateway | 14:35:22.135–24.540 | `Root=1-6678abcd-1a2b3c4d5e6f7a8b9c0d1e2f` | Completion request routed to Bedrock |
| (c) Bedrock | 14:35:24 | `3f8c1a9b-2d4e-4f61-8a9b-0c1d2e3f4a5b` | InvokeModel, 250 in / 180 out tokens |

## CloudWatch Logs Insights Queries

To query planes (b) and (c) in a real deployment:

### Query Plane (b): All Bedrock calls by an IAM role in a time window

```
fields @timestamp, @message, @logStream
| filter ispresent(role) and role like /fedllm-dev-app/
| filter @timestamp >= "2026-07-04T14:35:00Z" and @timestamp <= "2026-07-04T14:36:00Z"
| stats count() as invocation_count by @timestamp, role
```

Adjust the role ARN pattern and time window as needed. The `role` field name is illustrative — which fields the gateway emits depends on the LiteLLM logging configuration in `config.yaml`; adapt the filter to the fields actually present.

### Query Plane (c): Bedrock invocation metadata by model and identity

```
fields schemaType, timestamp, modelId, identity.arn, input.inputTokenCount, output.outputTokenCount
| filter schemaType = "ModelInvocationLog"
| filter identity.arn like /fedllm-dev-app/
| filter modelId = "anthropic.claude-sonnet-4-5-20250929-v1:0"
| stats sum(input.inputTokenCount) as total_input_tokens, sum(output.outputTokenCount) as total_output_tokens, count() as invocation_count by identity.arn
```

Run this over the `/aws/fedllm-dev/bedrock-invocations` log group to see aggregated token usage by role.

### Query Plane (c): Look for a specific invocation by time and role

```
fields @timestamp, requestId, identity.arn, modelId, input.inputTokenCount, output.outputTokenCount
| filter ispresent(identity.arn) and identity.arn like /fedllm-dev-app/
| filter @timestamp >= "2026-07-04T14:35:00Z" and @timestamp <= "2026-07-04T14:36:00Z"
| sort @timestamp desc
```

This returns all Bedrock invocations for that role in a 1-minute window, showing the `requestId` for manual correlation to plane (b).

## Summary

- **Planes (a) and (b) link via X-Amzn-Trace-Id** (if LiteLLM is configured to emit it).
- **Planes (b) and (c) link via timestamp + identity + model**, with the caveat that concurrent invocations may not be deterministically separable without additional context.
- **For deterministic correlation**, capture Bedrock's `requestId` in plane (b) and include it in plane (c) logs (requires full-content logging or custom instrumentation).
- **Query paths differ by plane:** CloudWatch Logs Insights covers planes (b) and (c); plane (a) lives in S3 (alb-logs bucket) and is queried with Athena or S3 Select.
- **Timing precision:** ALB and container logs carry sub-second timestamps; the Bedrock log record's `timestamp` is second-granular ISO 8601. Budget a 1–2 second window for cross-plane matching.
